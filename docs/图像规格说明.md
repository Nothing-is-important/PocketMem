## 图 2：Agent 执行流程图（核心 🔴）

**文件名：** `images/02-agent-execution-flow.png`
**插入位置：** 文档 §2 Agent 执行流程

**顶部：** 用户输入气泡 "张三推荐了哪家火锅店？"

**步骤 1：** "🔌 pre_route hook" — 灰色小框，标注"无注册 hook，直接通过"

**步骤 2：** "🎯 Router 意图分类" — 蓝色框
- 框内文字："LLM → 'memory_lookup'"
- 下方标注："关键词兜底：匹配到'张三' → 确认 memory_lookup"
- 右下角小标签："1 次 LLM 调用"

**步骤 3：** "📊 pre_retrieve hook" — 灰色小框
- 框内文字："UserProfile.inject_context()"
- 标注："注入：'用户常查询「张三」(23次)'"

**步骤 4：** "🔍 Retriever 混合检索" — 绿色大框
- 框内 4 步水平排列，箭头连接：
  `提取实体` → `ChromaDB 向量 + BM25` → `RRF 融合(k=60)` → `时间衰减`
- 最右侧标注结果："10 条结果"
- 标注："0 次 LLM 调用（纯检索）"

**步骤 5：** "⚖️ MemoryJudge 证据评估" — 橙色框
- 框内文字："组合评分 = 0.5×语义(0.81) + 0.2×时间(0.95) + 0.1×实体(0.2) + 0.2×重要性(0.7)"
- 下方三行小字：
  - "阈值过滤：0.76 ≥ 0.3 ✓"
  - "去重：10 → 8 条"
  - "充足性：8 ≥ 3 ✓ → sufficient=True"
- 标注："0 次 LLM 调用（纯数学）"

**步骤 6：** "✨ Generator 答案生成" — 紫色框
- 框内文字："构建 prompt（System + UserContext + Query + 5 个记忆片段）→ LLM 生成"
- 输出气泡："张三推荐了渝味火锅，地址在朝阳区建国路88号"
- 标注："1 次 LLM 调用"

**步骤 7：** "💾 post_generate hook" — 灰色小框
- 框内文字："UserProfile.record_query()"
- 标注："更新画像 → 保存 JSON"

**底部：** 汇总条
- "总计：2 次 LLM 调用（Router + Generator）"
- "延迟：Router ~2s + Retriever ~15ms + Judge ~0ms + Generator ~8-20s"

## 图 5：时间衰减曲线（补充 🟡）

**文件名：** `images/05-temporal-decay.png`
**插入位置：** 文档 §2.5.10 时间衰减
**用途：** 解释指数衰减 vs 线性衰减

### 画面描述

一张双曲线图：

- X 轴：时间（天），从 0 到 180
- Y 轴：记忆权重，从 0 到 1.0
- **蓝色曲线（指数衰减）：** decay = exp(-λ × days)，λ = ln(2)/30
  - 标注几个关键点：30天 ≈ 0.5，60天 ≈ 0.25，90天 ≈ 0.125，180天 ≈ 0.016
- **红色虚线（线性衰减）：** 在 120 天处权重变为 0
  - 标注："❌ 线性衰减：120 天后权重 = 0 → 永远找不到"

- 图下方标注："艾宾浩斯遗忘曲线 —— 人类记忆的遗忘规律是指数级的"
- 30 天处画一根虚线标注："半衰期 = 30 天（可配置参数）"

### 配色
- 指数曲线：蓝色实线
- 线性对比：红色虚线
- 半衰期标注：灰色虚线

## 图 6：LangGraph 状态图（补充 🟡）

**文件名：** `images/06-langgraph-stateflow.png`
**插入位置：** 文档 §2.5.2 LangGraph 介绍后
**用途：** 解释 StateGraph 的核心概念

### 画面描述

一张图解释三个核心概念：

**顶部：StateGraph**
- 一个矩形框代表 AgentState，标注每个字段：query, intent, retrieval_results, memory_context, final_answer...
- 标注："全局状态 —— 所有节点可读写，自动合并"

**中部：Node（节点）**
- 4 个节点水平排列：Router / Retriever / Judge / Generator
- 每个节点标注输入输出
- Router 输入：state["query"] → 输出：state["intent"] = "memory_lookup"

**底部：Edge（边）**
- 普通边：实线箭头（A 之后一定到 B）
  - Retriever → Judge：实线
- 条件边：虚线箭头 + 菱形判断符号（根据 state 决定去哪）
  - Router → [判断] → Retrieve 或 Generate
  - Judge → [判断] → Generate 或 Reflect（回到 Retrieve）

### 设计风格
- 简洁的流程图风格
- 实线 = 普通边，虚线 = 条件边

## 图 7：MemoryJudge 组合评分拆解（补充 🟡）

**文件名：** `images/07-composite-scoring.png`
**插入位置：** 文档 §8 MemoryJudge
**用途：** 面试时解释组合评分公式

### 画面描述

一个环形图/饼图 + 评分计算流程：

**饼图（权重分配）：**
- 语义相似度：50%（蓝色，最大块）
- 时间衰减：20%（橙色）
- 重要性评分：20%（绿色）
- 实体加权：10%（紫色，最小块）

**计算流程（水平排列）：**
1. 语义 = vector_score（余弦相似度）→ 0.81
2. 时间 = temporal_decay（指数衰减）→ 0.95
3. 实体 = entity_boost（人名/主题匹配）→ +0.2
4. 重要性 = importance（日期/地点/电话检测）→ 0.7

→ composite = 0.5×0.81 + 0.2×0.95 + 0.1×0.2 + 0.2×0.7 = 0.76

**底部对比：**
- 原始 Evidence Judge：~12 次 LLM 调用，~5s
- MemoryJudge：0 次 LLM 调用，<1ms
- 标注："通过数学公式替代 LLM 验证"

## 图 8：Hook 系统挂载点（补充 🟢）

**文件名：** `images/08-hook-system.png`
**插入位置：** 文档 §2.7 Agent 工程特性
**用途：** 可视化 Hook 的 4 个挂载点

### 画面描述

垂直时间线，Agent 执行从左到右的流程：

```
用户查询
   │
   ▼
[pre_route]     ← 挂载点 1：查询预处理
   │
   ▼
  Router
   │
   ▼
[pre_retrieve]  ← 挂载点 2：检索前（用户画像注入）
   │
   ▼
  Retriever
   │
   ▼
[post_retrieve] ← 挂载点 3：检索后（时间衰减、过滤）
   │
   ▼
  Judge → Generator
   │
   ▼
[post_generate] ← 挂载点 4：生成后（记录查询、更新画像）
   │
   ▼
  返回答案
```

右侧标注每个挂载点已注册的 hook：
- pre_route：（空）
- pre_retrieve：inject_user_profile ✅
- post_retrieve：apply_temporal_decay ✅
- post_generate：record_query ✅

底部标注："Hook 失败不阻塞主流程 —— 用户画像更新失败不应导致查询失败"

## 图片格式要求

- 格式：PNG（支持透明背景）
- 分辨率：至少 1200px 宽，保持 16:9 或 4:3 比例
- 背景：白色或透明
- 字体：中文用 Noto Sans SC 或微软雅黑，英文用 Inter 或 system font
- 风格：统一使用扁平化设计，避免 3D 效果
- 不要使用 emoji（部分渲染环境不支持），用简单的几何图标代替
