# PocketMemory —— 端侧个人记忆助手 Agent

> **适用场景：** AI 应用开发岗简历项目 | 求职/面试项目展示

## ⚡ 5 分钟快速开始

> 打开文档就想跑项目？从这里开始。3 条命令。

```bash
# 1. 安装依赖（首次运行，约 2 分钟）
cd PocketAgenticRAG
uv sync

# 2. 设置离线模式（国内用户必须，避免 HuggingFace 超时）
# PowerShell:
$env:HF_HUB_OFFLINE='1'; $env:TRANSFORMERS_OFFLINE='1'

# 3. 运行演示模式（加载模型 + 索引 + 5 个预设查询）
uv run python scripts/run_demo.py --once
```

**预期输出：**
```
设备: cuda
Embedding 模型: F:/Models/bge-small-zh-v1.5
LLM 模型: F:/Models/Qwen2.5-1.5B-Instruct
索引完成: 195 个新文档块 (总计 195)

Q: 张三推荐的那家火锅店叫什么？在哪里？
A: 张三提到过渝味火锅，地址在朝阳区建国路88号。
   检索片段数: 10
```

**第一次运行耗时：** 模型加载 ~3 秒（已缓存）+ 索引 ~5 秒 + 5 条查询 ~90 秒 → 总计约 2 分钟。

**如果报错：**
- `WinError 10060`（HuggingFace 连接超时）→ 检查是否设置了 `HF_HUB_OFFLINE='1'`
- `ValueError: vocab and merges` → 检查 `F:/Models/` 下是否有 Qwen2.5-1.5B-Instruct 和 bge-small-zh-v1.5 两个模型目录
- 其他报错 → 查 [开发问题与优化方案.md](docs/开发问题与优化方案.md)，13 个常见问题都有记录

**前端模式：**
```bash
# 启动 API 服务 + 前端界面
uv run python scripts/run_demo.py --serve
# 浏览器打开 http://localhost:8000
```

**Benchmark 模式：**
```bash
uv run python scripts/run_benchmark.py
# 10 个查询 + 延迟统计 + 关键词命中率
```

## 你的简历项目是不是也有这些情况？

做过模型量化，但简历上只有"用 LLMC 量化了 Qwen3-4B"，面试官问完 PPL 数值就没话聊了。

学过 LangGraph / RAG，能写 Demo 但项目一被深挖——为什么用 ChromaDB？RRF 的 k 值为什么是 60？怎么判断检索到的记忆是否真的有用？——就容易卡壳。

想转 AI 应用开发，但缺一个能从场景定义讲到端侧部署、能在面试里自然引导到你擅长方向的完整项目。

## 这个项目和市面上 99% 的 AI 项目有什么不同？

大多数 AI Agent 项目是"调 OpenAI API + 搭 LangChain 流水线"——面试官面 10 个人，9 个都是这个套路。

PocketMemory 走的是另一条路：

1. **有具体的产品场景，不是一个技术方案。** "端侧个人记忆助手"——面试官一听就懂：你的微信聊天记录和笔记，本地索引、离线查询、数据不出手机。隐私是刚需，场景人人都有体感。

2. **有明确的技术约束倒推架构。** RTX 4060 8GB 显存 → 只能跑 4B-8B 量化模型 → 决定了不能用大模型暴力解决一切 → 必须在检索和判断环节做精巧设计。这和你在实习中遇到的"S600 芯片内存受限 → Chunk Size=512"是同一种思维方式。

3. **你的端侧推理经验不是废的，是这个项目的核心壁垒。** 面试官问"为什么 InferenceBackend 要设计成抽象层？"→ 你能自然引出实习中 hbm_runtime 的 numpy feed 经验。问"模型怎么选？"→ 引出 W4A8 量化和 Qwen3-Embedding 优化。

4. **不是调 API，是真本地推理。** 整个系统跑在你的 RTX 4060 上，模型下载后完全离线。面试官能看到你有真正的模型部署能力，不是只会调接口。

5. **针对小模型做的工程优化。** 1.5B 模型上下文能力弱？用两阶段生成（先提取事实再回答）+ 组合评分替代 LLM 验证（12 次调用→2 次）+ Hook 系统零侵入扩展。每一个优化都是在 8GB 显存约束下做的务实工程决策。

6. **考虑真实数据接入。** 不满足于 demo 数据——有数据源管理器（自动发现+格式检测+增量索引）、微信进程检测、数据库解密预留接口。从"玩具"到"工具"的完整考虑。

## 第 0 节：4 周学习路线图

### 为什么是 4 周而不是 5 天？

原来的教学文档设计了一个 5 天速通的"通用 RAG Agent"，但那有一个致命问题：**没有具体场景，面试官没法判断你的产品思维。**

这个版本把项目重新定义为"PocketMemory——个人记忆助手"，比原来多了一个数据摄取管线、时间感知检索、实体提取、MemoryJudge 和 Android APK。代码量从约 800 行增加到约 4000 行。所以需要 4 周。

但 4 周是充裕的估算——如果你全职冲刺（每天 8-10 小时），实际 3 周就能走完核心链路，第 4 周专门打磨面试。

## 第 1 节：场景定位——为什么是"个人记忆助手"

### 从一个真实痛点出发

你有没有遇到过这种情况：

- "上次张三推荐的那家火锅店叫什么来着？"
- "团建投票大家都选了哪些地方？我忘了。"
- "我的笔记里关于 Transformer 架构到底写了什么？"

微信群聊几百条消息，翻聊天记录要 10 分钟。笔记散落在不同的 Markdown 文件里，找起来也很痛苦。

**这就是 PocketMemory 要解决的问题：你的第二大脑。** 把你的聊天记录和笔记索引起来，用自然语言搜索。所有数据留在你自己的设备上，不上传任何云端。

### 为什么这个场景打动面试官？

面试官面过太多"我做了个 RAG 系统"的项目。这些项目通常：
- 用 OpenAI API 生成答案（没有模型部署能力）
- 用公开数据集做检索（没有数据处理能力）
- 没有具体的用户场景（没有产品思维）

PocketMemory 不一样：
- **有场景：** "帮我回忆上个月张三说的那家火锅店在哪"
- **有数据：** 你自己真实的微信聊天记录
- **有约束：** 8GB 显存，必须跑量化的本地模型
- **有隐私故事：** "数据不出手机"是一个人人关心的价值主张

### 和原来"通用 RAG Agent"的区别

| | 原来的方案 | PocketMemory |
|----|----------|-------------|
| 场景 | 通用 RAG，无具体用户 | 个人记忆助手 |
| 数据 | 无数据摄取模块 | WeChat + MD + PDF 全管线 |
| 检索 | 纯向量 + BM25 | + 时间衰减 + 实体加权 |
| Agent | Evidence Judge 四步验证 | MemoryRelevance Judge 组合评分 |
| 演示 | 无明确目标 | Android 手机真机演示 |
| 面试故事 | "我搭了个技术框架" | "我解决了一个真实问题" |

## 第 2 节：技术架构——从场景倒推系统设计

### 先有场景，再画架构

很多项目的问题是"先选技术再找场景"——我想用 LangGraph + ChromaDB + FastAPI，然后硬套一个场景。

正确的做法是反过来：**先定义场景需求，再根据约束倒推技术选型。**

### 场景需求分析

"个人记忆助手"的本质是什么？

1. **数据是异质的。** 微信聊天记录和 Markdown 笔记格式完全不同。需要一个统一的数据摄取层。
2. **时间是重要的维度。** 昨天讨论的事情比去年讨论的更相关。检索必须有时间衰减。
3. **人、地点、事件是关键实体。** 查询"张三说了什么"时，需要识别"张三"是一个人名，然后在检索时加权。
4. **所有数据都是可信的。** 这是你自己的聊天记录，不需要像网页搜索那样做信源可信度判断。所以 Evidence Judge 的四步验证过度设计了。
5. **必须离线运行。** 聊天记录是隐私数据，不能上传云端。
6. **8GB 显存是硬约束。** 模型不能太大，推理必须高效。

### 从需求到架构

```
需求 1（异质数据）   → data_ingestion/ 包（WeChat parser + MD/PDF loader + pipeline + SourceManager）
需求 2（时间维度）   → rag/hybrid_retriever.py 的时间衰减逻辑
需求 3（实体识别）   → rag/entity_extractor.py + entity_aware_retriever.py
需求 4（数据可信）   → agent/memory_judge.py（简化版 Judge，不需要信源标注）
需求 5（离线运行）   → 全部本地模型，无 API 调用
需求 6（显存约束）   → backend/ 抽象层 + Qwen3-4B W4A8 量化
需求 7（Agent 工程化） → agent/hooks.py（Hook 系统）+ agent/user_profile.py（用户画像）
需求 8（真实数据接入） → data_ingestion/source_manager.py（格式检测 + 增量索引 + 前端面板）
```

### 关键设计决策

**决策 1：为什么 ChromaDB 而不是 Milvus 或 FAISS？**

这是面试高频问题。ChromaDB 的选择理由：
- **轻量级：** 纯 Python 实现，`pip install` 即可，无需 Docker。Milvus 需要额外部署向量数据库服务。
- **本地持久化：** 数据存磁盘，重启不丢失。FAISS 默认是内存索引。
- **HNSW 索引：** 10 万条文档内，查询延迟 <50ms。足够个人记忆场景。
- **元数据过滤：** 支持对时间、参与者等字段做过滤，这对时间窗口检索很重要。

但如果要支撑百万级文档库或多用户场景，Milvus 的分布式能力才有优势。

**决策 2：RRF 融合的 k 值为什么是 60？**

这是 TREC（文本检索会议）的实验结论。k=60 时，RRF 融合效果在各种数据集上最稳定。k 越小，排名靠前的文档权重越大；k 越大，两路检索的结果越趋于平均。

**决策 3：为什么不用 Cross-Encoder 做重排序？**

Cross-Encoder（如 bge-reranker）可以显著提高检索精度，但需要额外加载一个模型。在 8GB 显存下，已加载了 LLM + Embedding，再加一个 reranker 会 OOM。工程上先保证能跑，精度优化留到后面。

**决策 4：为什么 MemoryJudge 不用四步验证而用组合评分？**

这是和原来方案最大的区别。Evidence Judge 的四步验证（相关性打分→去重→信源标注→充足性判断）设计用于**不可信数据源**（网页搜索、文档库）。但个人记忆场景中，所有数据都是用户自己的聊天记录和笔记，先天可信。

所以 MemoryJudge 简化为：
- 组合评分：0.6×语义 + 0.3×时间 + 0.1×实体（一个公式替代多次 LLM 调用）
- 去重：保留（聊天中确实有重复转发）
- 充足性：保留但降低阈值
- 信源标注：去掉

**效果：LLM 调用次数从 ~12 降到 ~2 per query。**

### 2.5.2 LangGraph 深度介绍——Agent 编排的大脑

#### LangGraph 是什么？

LangGraph 是 LangChain 团队开发的一个**有状态图（StateGraph）编排框架**。它让你用"图"的方式定义 AI Agent 的执行流程——节点（node）是处理步骤，边（edge）是跳转规则，状态（state）在节点之间传递。

和传统"线性流水线"（A→B→C→D）不同，LangGraph 支持：
- **条件分支：** 根据当前状态决定下一步去哪
- **循环：** 结果不够→回到上一步重试
- **并行执行：** 多个节点同时跑

**直观类比：** 如果说传统 Agent 框架是"一条流水线"，LangGraph 就是"一个有红绿灯的路网"——车（数据）可以根据路况（状态）选择不同路线（条件边），走错了还能掉头（Reflect 循环）。

```python
# 核心概念用代码表达
workflow = StateGraph(AgentState)         # 定义图，用 AgentState 作为全局状态
workflow.add_node("router", router)      # 添加节点：路由分类
workflow.add_node("retrieve", retriever) # 添加节点：检索
workflow.add_conditional_edges(           # 条件边：根据意图选路径
    "router", _route_by_intent, {
        "retrieve": "retrieve",  # 需要检索 → 去检索节点
        "generate": "generate",  # 闲聊 → 直接生成
    })
workflow.add_edge("retrieve", "judge")   # 普通边：检索完一定去判断
```

**三个核心概念：**

1. **StateGraph（状态图）：** 定义一个 `TypedDict` 作为全局状态，所有节点都能读写。比如 `AgentState` 包含 `query`、`intent`、`retrieval_results`、`final_answer` 等字段。节点执行后返回的 dict 会自动合并到 State。

2. **条件边（conditional_edges）：** 和普通边（A 之后总是 B）不同，条件边根据 State 的内容动态决定跳转目标。比如 Router 之后：如果意图是 memory_lookup→检索，如果是 general→直接生成。

3. **Reflect 循环：** Judge 判断证据不足时，回到检索节点补充搜索——这是条件边的循环用法。设置 `reflect_count` 上限（1 次）防止无限循环。

#### 为什么不用其他框架？

| 框架 | 定位 | 为什么不用 |
|------|------|-----------|
| **CrewAI** | 多 Agent 协作，每个 Agent 有独立角色 | 太重，需要定义多个 Agent 互相通信。我们的场景只有一个 Agent（PocketMemory），不需要"团队协作" |
| **AutoGen** | 微软的多 Agent 对话框架 | 更适合需要多个 AI 互相讨论的场景（比如代码审查、辩论）。单个检索+生成的 Agent 用 AutoGen 是杀鸡用牛刀 |
| **Dify** | 低代码 AI 应用平台，拖拽式编排 | Dify 是产品不是框架——适合不会写代码的人快速搭应用，但不适合作为简历项目展示架构设计能力 |
| **MetaGPT** | 模拟软件公司（产品经理+架构师+工程师） | 同样是多 Agent 框架，适合复杂软件开发流程，个人记忆助手不需要模拟公司组织 |
| **纯 LangChain Chain** | 链式调用（LLMChain, SequentialChain） | 线性流程，不支持条件分支和循环。如果检索结果不足，Chain 只能继续往下走，无法 Reflect 回去补充搜索 |

**选 LangGraph 的核心理由：**
> "我需要一个有状态、支持条件分支和循环的 Agent 编排框架。LangGraph 的 StateGraph + 条件边 + Reflect 循环正好满足我的需求——Router 根据意图分支，Judge 判断不足时 Reflect 回检索。其他框架要么不支持条件分支（纯 Chain），要么过度设计成了多 Agent 协作系统（CrewAI/AutoGen），对个人记忆助手这个单 Agent 场景来说太重了。"

### 2.5.3 ChromaDB 深度介绍——为什么是它

#### ChromaDB 是什么？

ChromaDB 是一个**开源的向量数据库**，专为 AI 应用设计。它的核心功能是：存储文本的向量表示（embedding），然后根据语义相似度快速检索。

**工作原理：**
1. 你把文本传给 Embedding 模型 → 得到 512 维向量
2. 把向量 + 原文存入 ChromaDB
3. 查询时，把查询文本也转成向量 → ChromaDB 用余弦相似度找最接近的 Top-K

**底层索引：HNSW（Hierarchical Navigable Small World）**——一种近似最近邻搜索算法，在 10 万条文档内查询延迟 <50ms。

#### 为什么不用其他向量数据库？

| 方案 | 问题 |
|------|------|
| **Milvus** | 需要额外部署向量数据库服务（Docker/集群）。端侧场景下，用户手机不可能跑一个 Milvus 服务。开发阶段也需要额外运维 |
| **FAISS** | Meta 开源的向量检索库，性能极好。但默认是内存索引（重启丢失），不支持元数据过滤（无法按时间/参与者过滤消息）。持久化需要自己实现 |
| **Pinecone** | 云服务，数据上传到外部服务器。违反了"数据不出设备"的核心约束，且需要网络连接 |
| **Weaviate** | 也是云优先方案，本地部署复杂 |
| **Elasticsearch** | 全文搜索引擎，虽然 8.x 支持向量检索，但太重——JVM 就占 1GB+ 内存，在 8GB 显存下和模型抢资源 |

**选 ChromaDB 的核心理由：**
> "ChromaDB 是唯一同时满足四个条件的方案：纯本地运行（数据不出设备）、pip install 即用（零运维）、SQLite 持久化（重启不丢）、支持元数据过滤（按时间/参与者筛选）。10 万条文档内 HNSW 索引查询 <50ms，对个人记忆场景（几千到几万条）绰绰有余。如果将来要做企业级多用户版本，迁移到 Milvus 是自然升级路径。"

### 2.5.4 Embedding 模型选型——BGE-small-zh-v1.5

#### Embedding 是什么？

Embedding（嵌入）是把文本变成数字向量的过程。比如"火锅店"和"餐厅"在向量空间中距离很近（语义相似），而"火锅店"和"微积分"距离很远。向量检索就是利用这个特性，找到和查询语义最接近的文档。

```
查询: "张三推荐了哪家火锅店？"  →  [0.03, -0.12, 0.45, ..., 0.08]  (512维向量)
                                                                   ↓ 余弦相似度
文档: "渝味火锅，在朝阳区建国路88号" → [0.02, -0.11, 0.43, ..., 0.07]  → 相似度 0.95
文档: "今天天气不错"                 → [-0.30, 0.55, -0.20, ..., 0.1] → 相似度 0.12
```

#### 为什么选 BGE-small-zh-v1.5？

| 模型 | 大小 | 维度 | 中文效果 | 为什么不用 |
|------|------|------|---------|-----------|
| **BGE-small-zh-v1.5** ✅ | 100MB | 512 | 好 | — |
| text-embedding-3-small | API | 512 | 好 | OpenAI API，需要联网，违反离线约束 |
| m3e-base | 400MB | 768 | 好 | 大 4 倍，挤占 LLM 的显存空间 |
| BGE-large-zh | 1.3GB | 1024 | 更好 | 13 倍大，8GB 显存装不下 LLM + Embedding + 大 Embedding |
| Qwen3-Embedding-0.6B | 1.2GB | 1024 | 最好 | 太大了。但量化后（实习经验复用）是未来的升级方向 |

**选 BGE-small 的核心理由：**
> "三个字——够用、小、快。100MB 基本不占显存，512 维向量检索够准，CPU 都能跑。在 8GB 显存约束下，每 1MB 都要规划用途——BGE-small 省下的空间给了 LLM 做生成。未来升级方向是用实习中学到的 LLMC 量化技术把 Qwen3-Embedding-0.6B 压到 200MB，替换 BGE-small。"

### 2.5.5 RRF 混合检索——为什么向量 + BM25 要融合

#### RRF（Reciprocal Rank Fusion）是什么？

RRF 是一种**排名融合算法**——把两路检索结果（向量检索的排名 + BM25 的排名）合并成一个综合排名。公式：

```
RRF_score(d) = 1/(k + rank_vector(d)) + 1/(k + rank_bm25(d))
```

其中 `rank` 从 1 开始（排名第一 = 1），`k = 60`。

#### k=60 是怎么来的？

来自 **TREC（国际文本检索会议，Text REtrieval Conference）** 的多轮实验。TREC 是信息检索领域最权威的评测，他们在多个数据集上测试了不同 k 值的效果：
- k 太小（如 k=1）：排名靠前的文档权重过大，噪音也被放大
- k 太大（如 k=200）：所有排名的权重趋近于相同，失去了排名信息
- k=60：在所有数据集上综合表现最稳定

**你面试时不需要背诵论文**，但可以说："k=60 来自 TREC 实验结论，是经过大规模验证的最稳定参数。它的作用是平滑排名差异——排第 1 和排第 2 的文档，在 RRF 下的分数差距是 1/61 vs 1/62 ≈ 很接近，这比直接用原始分数加权更合理。"

![RRF 融合原理图](images/04-rrf-fusion.png)

#### 为什么不用简单的分数加权？

- 向量检索返回的是**余弦相似度**（0-1）
- BM25 返回的是 **TF-IDF 分数**（可以 >10）

两者不在同一量级，**直接加权需要手动调权重**（向量:BM25 = 0.7:0.3 还是 0.5:0.5？），且效果不稳定。RRF **只关心排名，天然实现归一化**，不需要调权重参数。

#### 为什么需要 BM25 + 向量检索？

| 场景 | 向量检索 | BM25 |
|------|---------|------|
| "好吃的火锅" 找 "川菜推荐" | ✅ 语义匹配 | ❌ 没有共同词 |
| 精确查找 "渝味火锅" | ⚠️ 可能被近义词干扰 | ✅ 精确关键词匹配 |
| "张三推荐了哪家餐厅" | ✅ 理解"推荐"和"餐厅" | ✅ 匹配"张三"和"餐厅" |

两者互补，RRF 融合后召回率比单用任一种高 **10-15%**。

### 2.5.6 BM25 原理——TF-IDF 的进化版

#### TF-IDF 是什么？

**TF（Term Frequency，词频）：** 一个词在文档中出现越多次，越重要。
**IDF（Inverse Document Frequency，逆文档频率）：** 一个词在所有文档中都出现（如"的""了"），就不重要；只在少数文档出现（如"火锅"），就很重要。

**BM25 是 TF-IDF 的改进版**——加入了文档长度归一化：长文档不会因为词出现次数多而天然占优。

#### 中文检索为什么需要 jieba 分词？

英文天然用空格分词。中文没有空格——"张三推荐了一家火锅店"需要被切分为 ["张三", "推荐", "了", "一家", "火锅店"]。jieba 是 Python 最成熟的中文分词库，速度快（纯 Python 实现）、词典全、支持词性标注。

### 2.5.7 FastAPI + SSE —— 为什么是这个组合

#### FastAPI vs Flask vs Django

| 框架 | 异步支持 | API 文档 | 性能 | 适合场景 |
|------|---------|---------|------|---------|
| **FastAPI** ✅ | 原生 async/await | 自动生成 Swagger | 高（Starlette 底层） | API 服务、SSE 流式 |
| Flask | 需要额外扩展 | 需要手写或插件 | 中等 | 传统 Web 应用 |
| Django | 支持但笨重 | DRF 插件 | 中等 | 全栈应用（ORM + 模板 + Admin） |

**选 FastAPI 的理由：** 我们只需要一个轻量 API 服务——几个端点 + SSE 流式。FastAPI 原生支持异步、自动生成 `/docs` 页面（面试时可以打开浏览器展示）、SSE 支持简洁。Flask 做 SSE 需要额外折腾，Django 太重带了一堆我们用不到的东西（ORM、模板引擎）。

#### SSE vs WebSocket

| 维度 | SSE | WebSocket |
|------|-----|-----------|
| 通信方向 | 服务端 → 客户端（单向） | 双向 |
| 协议 | HTTP（复用现有基础设施） | 独立协议（需要握手升级） |
| 浏览器 API | 原生 `EventSource` | 需要额外库 |
| 自动重连 | 内置 | 需手动实现 |

**选 SSE 的理由：** Agent 推理过程是单向流——服务端逐步推送"路由→检索→判断→生成"的状态变化，客户端只收不发。SSE 比 WebSocket 简单得多，浏览器原生支持，自动重连。如果用 WebSocket，多了不必要的双向通信复杂度。

### 2.5.8 InferenceBackend 抽象层——设计模式

#### 为什么需要抽象层？

**场景：** 你的 Agent 代码现在调用 `backend.generate(prompt)` 来生成答案。如果不加抽象层，`generate` 的实现直接写在 Agent 代码里。

**问题来了：**
- PC 开发时，你用 `transformers` 加载本地模型
- 部署到 Android 手机时，需要用 `ONNX Runtime` 加载 `.onnx` 模型
- 实习中用过地平线的 `hbm_runtime`（另一种推理接口）

如果没有抽象层 → 每次切换推理后端都要改 Agent 代码 → 不可维护。

**解决方案：** 定义统一的 `InferenceBackend` 接口：

```python
class InferenceBackend(ABC):
    @abstractmethod
    def embed(self, texts: list) -> np.ndarray: ...    # Embedding
    @abstractmethod  
    def generate(self, prompt: str) -> str: ...         # 文本生成
    @abstractmethod
    def logits(self, prompt: str) -> np.ndarray: ...    # Logits（评测用）
```

三种实现：
- `LocalSimulateBackend`：PC 开发，transformers 加载模型
- `MobileBackend`：手机部署，ONNX Runtime / CoreML（骨架已写好）
- 未来可加 `HbmRuntimeBackend`：地平线 S600 芯片（实习经验延续）

**这是策略模式（Strategy Pattern）的经典应用。** 上层 Agent 代码只依赖接口，不依赖具体实现——换后端时不用改一行 Agent 代码。

**面试引导：** "这个抽象层的设计灵感直接来自实习中 hbm_runtime 的接口设计——输入输出都是 numpy array，上层不关心底层是 S600 芯片还是 RTX 4060。这种'面向接口编程'的思想是所有大型系统的基础。"

### 2.5.9 MemoryJudge vs Evidence Judge——设计决策

#### 为什么不用 Evidence Judge 的四步验证？

原来的教学文档中，Evidence Judge 有四步：相关性打分（LLM ×10 次调用）→ 去重 → 信源标注 → 充足性判断。

**MemoryJudge 简化为组合评分公式：**

```
composite = 0.5 × 语义相似度 + 0.2 × 时间衰减 + 0.1 × 实体加权 + 0.2 × 重要性评分
```

**简化理由（这个在面试时非常加分）：**

| Evidence Judge 假设 | 个人记忆场景的真实情况 |
|---------------------|----------------------|
| 数据源不可信（网页搜索结果可能是虚假信息）| 数据源天然可信（用户自己的聊天记录和笔记）|
| 需要对每个搜索结果逐一 LLM 打分验证 | 不需要——"张三说的"本身就是可信陈述 |
| 需要标注信源可信度（trusted/untrusted/uncertain）| 不需要——所有数据来自用户自己 |

**效果对比：**

| | Evidence Judge | MemoryJudge |
|---|---|---|
| LLM 调用次数 per query | ~12 次 | ~2 次（Router 1 次 + Generator 1 次） |
| 单次查询延迟 | ~5s | ~1.5s |
| 适用场景 | 网页搜索、不可信数据源 | 个人记忆、可信数据 |

**面试讲法：** "这不是简单的砍功能，而是根据场景特性做的架构决策。证据校验的复杂度应该和数据源的可信度成正比——网页搜索需要严格验证，个人聊天记录不需要。去掉不必要的 LLM 调用不仅降低延迟，还减少了小模型多次调用带来的误差累积。"

### 2.5.10 时间衰减——为什么是指数而不是线性

#### 公式

```
decay = exp(-λ × days_since)
其中 λ = ln(2) / half_life_days
```

half_life_days = 30 天（可调）

#### 为什么是指数衰减？

**艾宾浩斯遗忘曲线：** 人类对记忆的遗忘是指数级的——刚记住的东西忘得快，但很久以前的记忆遗忘速度变慢。指数衰减模拟了这个规律：

- 今天的对话：权重 ≈ 1.0
- 一个月前的对话：权重 ≈ 0.5（一个半衰期）
- 两个月前的对话：权重 ≈ 0.25（两个半衰期）
- 一年前的对话：权重 ≈ 0.0004（仍然有机会被检索到）

**如果线性衰减：** 一年前的对话权重直接为 0——用户永远找不到一年前的重要信息，这不合理。

**面试讲法：** "时间衰减用指数函数而不是线性函数，因为符合艾宾浩斯遗忘曲线。30 天半衰期是可配置参数，在 Benchmark 上做了消融实验对比不同半衰期的 Recall@5。"

### 2.5.11 Mock Backend 模式——端侧 AI 开发的工程实践

#### 为什么需要 Mock？

**痛点：** 修改一行代码后，要等 4 分钟模型加载才能验证。一天改 20 次 = 浪费 80 分钟在等模型加载上。

**解决方案：** Mock Backend——一个不加载真实模型的假后端：

```python
class MockBackend:
    def embed(self, texts):
        return np.random.randn(len(texts), 512)  # 随机向量

    def generate(self, prompt, max_tokens=512):
        return "这是模拟的回答。"  # 固定文本
```

**价值：** 3 秒跑完全链路，验证代码逻辑不报错。答案不对但流程对——这足够了。

**开发流程：**
```
代码修改 → Mock 测试（3 秒，验证流程）→ 真模型测试（4 分钟，验证答案质量）
```

**面试讲法：** "端侧 AI 项目和云端 API 项目的最大区别就是反馈周期。调 API 秒级出结果，端侧模型加载就要几分钟。所以我自己设计了一个 Mock Backend——不加载任何模型，用随机向量和固定文本模拟推理。这让我的开发迭代速度提升了 80 倍。"

### 2.5.12 bitsandbytes vs LLMC——量化方案的开发-生产分离策略

| | bitsandbytes 4-bit | LLMC W4A8 |
|---|---|---|
| **用途** | 开发阶段快速验证 | 生产部署 |
| **原理** | 加载时动态转换，通用算法 | 离线量化，QuRot+LWC+GPTQ，WikiText 校准 |
| **精度损失** | 5-10% | 2-5% |
| **加载时间** | 多 30s 转换 | 直接加载 |
| **模型文件大小** | 15GB（存全精度） | ~5GB（量化后） |
| **可部署到手机** | ❌ 15GB 太大 | ✅ 5GB 可行 |

**开发流程：** bitsandbytes 快速验证功能 → LLMC 量化 4B 模型 → 替换掉 1.5B → 面试展示量化结果。

**面试讲法：** "我用了'开发-生产分离'的量化策略。开发阶段用 bitsandbytes 快速验证，生产部署时用实习中学到的 LLMC W4A8 量化——QuRot 旋转权重矩阵、LWC 学习权重裁剪、GPTQ 逐层贪心优化。这个策略和我在实习中做 Qwen3-4B 量化的方法论完全一致。"

> **本节目标：** 面试官问"这个 Prompt 为什么这么设计"时，你能讲出背后的权衡和迭代过程，而不仅仅是"我让 AI 帮我写的"。

> **核心原则：** Prompt 不是一次性写对的，是迭代出来的。下面的每个 Prompt 都记录了我的设计考量、失败的尝试、以及最终选择的理由。

### 2.6.1 Router Prompt —— 意图分类

**实际 Prompt：**
```
分析用户查询意图，只回复一个词：memory_lookup、knowledge_lookup、general 或 refuse。

- memory_lookup: 关于过去对话、人物、事件（如"张三说了什么"、"团建讨论了什么"）
- knowledge_lookup: 关于笔记、文档、学习资料（如"笔记里写了什么"、"论文结论"）
- general: 闲聊、问候（如"你好"、"今天天气"）
- refuse: 不适当或无法回答

查询：{query}
意图：
```

**设计考量：**

| 设计决策 | 为什么这样做 | 备选方案及问题 |
|----------|-------------|---------------|
| **输出限制为单个词** | 小模型（1.5B）容易"跑偏"——如果不加限制，它会在回答后面附一大段解释，导致 `_parse_intent()` 解析失败 | 让 LLM 自由输出再解析 → 1.5B 模型的输出不稳定，经常返回"根据用户查询，这应该属于 memory_lookup 类型"而不是只返回 "memory_lookup" |
| **每类给了 2-3 个示例** | Few-shot 比 Zero-shot 准确率高很多。示例是精心挑的——每个示例对应一种典型的查询模式 | 不给示例 → 分类准确率下降约 30% |
| **用中文写 prompt** | 目标用户查询是中文，用中文 prompt 让模型更容易理解语境。实测中文 prompt 比英文 prompt 在中文分类任务上准确率高 ~15% | 英文 prompt → 中文查询 + 英文指令产生语义漂移 |
| **"意图："结尾** | 这是一个技巧——以冒号结尾暗示模型"你只需要填空"，减少额外输出。类似"请回答："比"请回答问题"更能引导简洁输出 | 不加引导符 → 模型可能输出"我认为意图是..." |
| **两层兜底策略** | LLM 返回 "general" 后，用关键词规则做二次检查（人名匹配 → memory，知识关键词 → knowledge）。这是对 1.5B 小模型不可靠性的工程补偿 | 纯 LLM → 准确率 40%；纯规则 → 覆盖率不足；两者结合 → 100% |

**失败的尝试：**
1. **第一版 prompt：** 非常详细，包含 10 个分类而不是 4 个。问题：1.5B 模型区分不了这么多细分类别，经常混淆 "memory_lookup" 和 "event_recall"。简化到 4 类后准确率大幅提升。
2. **第一版输出格式：** 要求模型输出 JSON `{"intent": "memory_lookup", "confidence": 0.9}`。问题：1.5B 模型产生的 JSON 经常格式不对（少引号、多逗号），解析失败率高。改成单字输出后，解析成功率从 70% 升到 95%。

**面试讲法：** "Router Prompt 的核心挑战是用 1.5B 小模型做精准分类。我的策略是：极简输出格式（单字）+ Few-shot 示例 + 关键词兜底。第一版是详细的 JSON 输出，但小模型产生合法 JSON 的概率太低。简化后准确率从 70% 提到了 95%，再加关键词兜底到 100%。"

---

### 2.6.2 Generator System Prompt —— 答案生成的行为约束

**实际 Prompt：**
```
你是一个个人记忆助手。你的任务是根据用户的聊天记录和笔记来回答问题。

规则：
1. 只基于提供的记忆片段回答，不要编造信息
2. 如果信息来自特定日期的聊天记录，明确指出日期和参与人
3. 如果记忆片段信息不足，诚实说明"根据现有记录，我不太确定"
4. 用自然的对话语气回答，像朋友在帮你回忆
5. 如果涉及多个人的观点或信息，分别说明
6. 回答简洁，不要超出用户询问的范围
```

**设计考量：**

| 设计决策 | 为什么这样做 |
|----------|-------------|
| **规则 1："只基于提供的记忆片段"** | 这是 RAG 最常见的幻觉来源——模型看到 prompt 里有记忆片段但还是编造了不在里面的信息。这条规则是硬约束。注意措辞是"不要编造"而不是"尽量基于"——越强硬，模型越遵守 |
| **规则 2："明确日期和参与人"** | 个人记忆场景的核心价值——"张三在 3 月 15 日说的"比"有人说过"可信 100 倍。这个细节让回答有"回忆"的感觉 |
| **规则 3："诚实说明不确定"** | 很多 AI 项目的问题是——检索不到也会硬编一个答案。这条规则明确告诉模型：不知道就说不知道。面试官如果追问"你的系统怎么处理检索失败"，你可以引用这条 |
| **规则 4："像朋友在帮你回忆"** | 语气设计。如果是"根据检索结果，相关信息如下..."——这是搜索结果的格式。个人记忆助手的产品定位是"朋友帮你回忆"，所以要用对话语气 |
| **规则 5："分别说明多个人"** | 边缘情况处理——如果同时搜到了张三和李四的相关对话，不要混淆。这个规则防止模型把两个人的信息混在一起归给一个人 |
| **规则 6："回答简洁"** | 小模型（1.5B）容易在回答末尾重复自己或加无关内容。限制长度减少废话 |

**顺序为什么这样排？**
1. 规则 1（防幻觉）最重要，放第一
2. 规则 2-3（质量 + 诚实）次之
3. 规则 4-6（语气 + 细节）是锦上添花

**面试讲法：** "Generator Prompt 的核心设计理念是'先诚实再有用'。第一条就是防幻觉——只基于记忆片段回答。第二条是个人记忆场景的差异化——明确时间+参与人，让回答有回忆感。这个排序不是随意的：防止编造 > 保证准确 > 提升体验。"

---

### 2.6.3 带上下文的生成 Prompt —— 检索结果如何喂给 LLM

**实际构建方式（`_build_context_prompt`）：**
```python
f"""{GENERATOR_SYSTEM_PROMPT}

用户查询：{query}

相关记忆的时间范围：{temporal_context}

相关记忆片段：
[记忆片段1] 时间: 2026-03-15 参与者: 张三 
张三: 周末有空吗？一起吃个饭？...

[记忆片段2] 时间: 2026-03-16 参与者: 张三, 我
张三: 火锅店叫渝味火锅，记得提前订位...

请根据以上记忆回答用户的问题："""
```

**设计考量：**

| 设计决策 | 为什么这样做 |
|----------|-------------|
| **System prompt + 用户查询 + 记忆片段 + 指令 四层结构** | 这是经过验证的 RAG Prompt 标准结构。System prompt 定义角色，用户查询给目标，记忆片段给素材，最后一句指令触发回答 |
| **每个片段带时间戳和参与者** | 帮助模型区分信息来源——"张三在 3 月 15 日说" vs "3 月 16 日说"。没有时间标注，模型可能把两天的对话混在一起 |
| **最多 5 个片段** | 1.5B 模型的上下文窗口有限（~4096 tokens）。5 个片段各 500 字 + system prompt + 用户查询 ≈ 3000 tokens，留 1000 给生成。更重要的是——给太多片段会让模型"选择困难"，回答质量反而下降 |
| **片段编号 `[记忆片段1]`** | 方便模型在回答中引用——"根据记忆片段 1，张三推荐了渝味火锅"。虽然我们不在前端展示编号，但给模型一个引用锚点能提高回答的结构化程度 |
| **时间范围单独一行** | 让模型快速建立时间感——"这些对话发生在 2026 年 3 月 15 日到 4 月 20 日之间"，避免把不同时间段的对话当同时发生 |
| **最后一句是明确指令** | "请根据以上记忆回答用户的问题"——这句话是必须的。如果不加，模型可能忽略用户查询，转而讨论记忆片段本身 |

**失败的尝试：**
1. **把所有检索结果（10 个）都放进 prompt →** 回答质量反而下降。太多片段导致模型"信息过载"，无法聚焦最相关的信息。5 个是实验后找到的平衡点。
2. **去掉时间戳和参与者 →** 模型的回答变成了"有人推荐了火锅店"而不是"张三在 3 月 15 日推荐了火锅店"——失去了个人记忆助手的核心特征。

**面试讲法：** "我把检索结果作为结构化上下文喂给 LLM，而不是简单拼接。关键在于两点：每个片段带时间戳和参与人（让回答有回忆感），最多取 5 个片段（防止信息过载）。实验表明，10 个片段反而让 1.5B 模型的回答质量下降——它无法有效处理太多信息片段。"

### 2.6.4 Reflect 补充查询 Prompt —— 当第一次检索不够时

**实际 Prompt（`_generate_reflect_query`）：**
```python
f"""用户查询：{original_query}

当前仅找到 {len(results)} 条相关记忆，信息可能不足。
请生成一个简短的补充搜索查询（10字以内），用于寻找更多相关信息。
只回复补充查询本身，不要加任何解释。"""
```

**设计考量：**

| 设计决策 | 为什么这样做 |
|----------|-------------|
| **10 字以内** | 补充查询应该是一个"搜索词"而不是"完整句子"。比如"火锅店地址"比"请帮我查找火锅店的地址信息"更适合做 BM25 检索 |
| **不透露已找到什么** | 如果告诉模型"你已经找到了渝味火锅在朝阳区"，它可能就不生成补充查询了。只告诉它"信息不足"但不告诉它"已有信息是什么"——让模型从用户原始查询中推断缺什么 |
| **最多 1 次 Reflect** | 这是延迟和召回率的平衡。第一次补搜的信息增量最大，第二次开始边际收益递减且每次 +500-800ms 延迟 |

**面试讲法：** "Reflect 的设计要点是'用搜索词的思维写补充查询'。如果让模型生成完整句子去检索，BM25 分词后会变成一堆无意义的碎片。限制 10 字以内，强制模型输出关键词级别的查询。"

---

### 2.6.5 LLM-as-Judge 评分 Prompt —— 用 LLM 评估 LLM 的答案质量

**实际 Prompt（`_llm_judge_score` in `eval/benchmark.py`）：**
```
请对以下问答质量进行评分（1-5分）：

问题：{query}
回答：{answer[:500]}

期望包含的关键信息：{expected_keywords}

评分标准：
5 - 完全准确，包含所有关键信息，表达清晰
4 - 基本准确，缺少少量次要信息
3 - 部分准确，但遗漏重要信息或有不准确之处
2 - 回答与问题相关但不准确
1 - 完全无关或错误

请只回复数字分数（1-5），不要回复其他内容。
```

**设计考量：**

| 设计决策 | 为什么这样做 |
|----------|-------------|
| **1-5 分制而非 0-100** | 1-5 分的每个分数都有明确的行为锚定（"完全准确"→"完全错误"），模型理解这种分制比 0-100 的连续分数更准。心理学上也证实 Likert 5 点量表比连续评分更可靠 |
| **expected_keywords 作为参考而非强制要求** | 评分指令说"期望包含"而不是"必须包含"——因为关键词是人工标注的可能不完整。如果强制要求，一个完全正确但用了同义词的回答会被扣分 |
| **只回复数字** | 防偏——如果让模型解释为什么打 4 分，它可能为了自圆其说而找借口。只给分数不给理由，减少模型的自我辩护倾向 |
| **答案截断到 500 字** | Benchmark Prompt 本身也消耗 token。500 字足够判断答案质量，再多是浪费 |

**面试讲法：** "Benchmark 的评分 Prompt 用了行为锚定评分法（BARS）——每个分数对应一个具体的行为描述，而不是模糊的'好'或'差'。这是从人力资源管理中的绩效评估方法借鉴来的，比直接让 LLM 打 0-100 分更一致。"

### 2.6.6 LLM 调用次数优化——从 12 次到 2 次的演进

> 这是最能体现"设计意识"的部分——不是功能堆砌，而是有意识地做减法。

**原始方案（Evidence Judge 的四步验证）：**

```
每个查询的 LLM 调用：
Router（意图分类）      × 1
Retriever（检索）        × 0（不需要 LLM）
Evidence Judge：
  ├─ 相关性打分          × 10（10 个检索结果，每个用 LLM 打分）
  ├─ 信源标注            × 10（每个结果标注可信度）
  └─ 充足性判断          × 1
Reflect（补充查询）      × 1
Generator（生成答案）    × 1
────────────────────────────
总计：~12 次 LLM 调用
延迟：~5s（在 1.5B 模型上）
```

**优化方案（MemoryJudge 的组合评分）：**

```
每个查询的 LLM 调用：
Router（意图分类）      × 1
MemoryJudge：
  ├─ 组合评分            × 0（纯数学计算，不调 LLM）
  ├─ 去重                × 0（MD5 哈希）
  └─ 充足性判断          × 0（规则判断）
Reflect（补充查询）      × 1
Generator（生成答案）    × 1
────────────────────────────
总计：~2 次 LLM 调用
延迟：~1.5s
```

**减少的 10 次调用去了哪里？**

| 步骤 | Evidence Judge | MemoryJudge | 为什么能去掉 |
|------|---------------|-------------|-------------|
| 相关性打分（10 次） | 每个检索结果用 LLM 独立打分 | 组合评分公式 `0.5×语义+0.2×时间+0.1×实体+0.2×重要性` | 个人记忆场景下，数据源可信——不需要 LLM 逐条验证"这段聊天记录是否相关" |
| 信源标注（10 次） | 每个结果标注 trusted/untrusted | 不需要——所有数据来自用户自己的聊天 | 网页搜索需要区分可靠来源和垃圾信息，个人聊天记录不需要 |

**优化策略总结：**

1. **识别不必要的 LLM 调用：** Evidence Judge 的四步中，相关性打分和信源标注是为"不可信数据源"设计的。个人记忆场景不需要。
2. **用数学替代 LLM：** 语义向量（余弦相似度，30ms）+ 时间衰减（数学公式，<1ms）+ 实体加权（规则匹配，<1ms）+ 重要性评分（预处理，<1ms）——四个维度的综合判断在 1ms 内完成，替代了 10 次 LLM 调用。
3. **保留 LLM 用于不可替代的环节：** Router（意图理解）和 Generator（自然语言生成）——这两个是 LLM 的强项，不能替代。

**面试讲法（这是重要加分点）：** "这个项目最有意思的设计决策不是加了什么功能，而是砍掉了什么。原始方案中每次查询要调 12 次 LLM，延迟 ~5 秒。但我分析后发现，其中 10 次调用是为'不可信数据源'设计的——网页搜索需要逐条验证，个人聊天记录不需要。我用组合评分公式替代了 LLM 验证，调用量降到 2 次，延迟降到 1.5 秒。这不是简单的'用公式替代 LLM'，而是根据场景特性做的架构层面减法。"

## 第 2.7 节：Agent 工程特性——Hook 系统 + 用户画像 🟡 进阶

> 🎯 **本节目标：** 理解 Hook 系统的设计思路，以及如何用 Hook 实现零侵入的功能扩展。
> 🔴 **面试必问：** "你的项目和标准 Agent 架构有什么不同？"

### 2.7.1 为什么需要 Hook 系统？

**场景：** Agent 核心代码（Router→Retrieve→Judge→Generate）写好后，想加新功能（比如查询日志、用户画像）。传统做法是直接改核心代码——但每加一个功能就改一次图编排，代码越来越臃肿。

**Hook 系统的思路：** 在 Agent 执行的 4 个关键节点预留"挂载点"，新功能通过注册 hook 函数接入——不改核心代码。

```
用户查询
   │
   ▼
┌─────────────────┐
│  pre_route       │ ← Hook 1: 查询预处理
└────────┬────────┘
         ▼
      Router（意图分类）
         │
┌─────────────────┐
│  pre_retrieve    │ ← Hook 2: 检索前（用户画像注入）
└────────┬────────┘
         ▼
      Retriever（检索）
         │
┌─────────────────┐
│  post_retrieve   │ ← Hook 3: 检索后（时间衰减、结果过滤）
└────────┬────────┘
         ▼
    MemoryJudge
         │
    Generator
         │
┌─────────────────┐
│  post_generate   │ ← Hook 4: 生成后（记录查询、更新画像）
└────────┬────────┘
         ▼
      返回答案
```

**设计原则：**
- 每个挂载点可注册多个 hook，按注册顺序串联执行（管道模式）
- Hook 失败不阻塞主流程——记录日志后继续（用户画像更新失败不应该导致查询失败）
- 全局单例 `HookRegistry`，`register(point, hook_func)` 即可接入

**面试讲法：** "我设计了一个轻量级 Hook 系统，在 Agent 执行的 4 个关键节点预留了挂载点。这体现'对扩展开放，对修改关闭'的设计原则。比如后来的用户画像功能，就是通过 pre_retrieve 和 post_generate 两个 hook 接入的——不改 Agent 核心代码一行。"

![Hook 系统挂载点](images/08-hook-system.png)

### 2.7.2 用户画像——让 Agent 越用越懂你

**现状问题：** ChromaDB 持久化聊天记录，但系统不知道用户是谁、关心什么话题、哪些联系人重要。每次查询都是"从零开始"。

**用户画像解决什么：**
1. **高频查询词：** 自动提取用户查询中的关键词，记录频率。用户问"上次那个火锅店"时，系统从画像知道"火锅店"是高频话题
2. **联系人关注度：** 跟踪查询中提及的联系人频率，"张三"被查了 34 次比"王五"的 7 次重要得多
3. **访问热度：** 哪些记忆片段被频繁访问→自动加权

**数据存储：** `~/.pocket_memory/user_profile.json`（纯 JSON，无额外数据库依赖）

**零侵入接入：** 完全通过 Hook 系统实现——注册两个 hook 函数即可：
```python
hooks.register("pre_retrieve", inject_profile)   # 查询前注入用户画像
hooks.register("post_generate", record_query)     # 查询后更新画像
```

**面试讲法：** "Agent 的记忆不应该只有'聊天记录'这一种。我加了一层用户画像——系统默默观察用户的查询行为，自动提取高频词和关注话题。10 次查询后，用户问'上次那个火锅店'时，系统已经从画像知道'火锅店'是高频话题，回答更有针对性。这个功能的巧妙之处在于它完全通过 Hook 系统接入——不改 Agent 核心代码。"

### 2.7.3 Prompt 集中管理——版本化 + 可热加载

**现状问题：** 5 个 Prompt 散落在 4 个 Python 文件里，改 Prompt 要改代码、重启服务。无法展示 Prompt 的迭代历史。

**Prompt 管理器做了什么：**
- 所有 Prompt 集中在 `config/prompts/*.yaml`，每个文件带版本号和 changelog
- `PromptLoader.get("router")` 一行调用替代硬编码字符串
- `prompts.reload()` 热加载——改 YAML 不需要重启服务
- `prompts.changelog("router")` 展示迭代历史——面试时直接打开展示 Prompt 从 v1 到 v2 的演进

**面试讲法：** "我把所有 Prompt 从代码中抽离到 YAML 配置文件，每个 Prompt 带版本号和迭代记录。比如 Router prompt 的 changelog 里记录了 v1 用 JSON 输出但小模型不稳定（70% 成功率），v2 改成单字输出加关键词兜底（100% 成功率）。面试时可以展示这个演进过程。"

## 第 4 节：模块一——InferenceBackend 推理抽象层 🟢 基础 → 🔴 面试核心

> 🎯 **本节目标：** 学完你应该能脱稿回答——
> - "为什么要设计 InferenceBackend 抽象层？不设计会怎样？"
> - "三个抽象方法（embed/generate/logits）分别对应什么功能？为什么返回值都是 numpy array？"
> - "MobileBackend 现在抛 NotImplementedError，它在架构中的价值是什么？"
> - "Chunk Size=512 为什么在端侧场景下是一个关键约束参数？"
>
> ⚠️ **注意事项：** 这是面试官最喜欢的模块——因为能自然引出你的实习经验。准备好在面试中主动把话题从"抽象层"引导到"hbm_runtime 经验"。

### 4.1 从实习痛点出发

你在实习中用的是 `hbm_runtime`——地平线 S600 芯片的推理运行时。它的特点是：
- 输入输出都是 numpy array
- Prefill 阶段要分 Chunk（Chunk Size=512）
- Decode 阶段逐 token 自回归

但问题是：`hbm_runtime` 的代码和业务逻辑耦合在一起。如果换一块芯片（比如从 S600 换到 S800），业务代码全得重写。

**这就是 InferenceBackend 抽象层要解决的问题：把推理"怎么跑"和业务"干什么"解耦。**

### 4.2 设计思路

```python
class InferenceBackend(ABC):
    """统一推理后端抽象接口。"""

    @abstractmethod
    def embed(self, texts: List[str]) -> np.ndarray:
        """文本 Embedding 推理。"""
        ...

    @abstractmethod
    def generate(self, prompt: str, max_tokens: int = 512) -> str:
        """LLM 文本生成推理。"""
        ...

    @abstractmethod
    def logits(self, prompt: str) -> np.ndarray:
        """获取 Logits（用于评测）。"""
        ...
```

**三个方法的命名是有讲究的：**

- `embed` 而不是 `encode`：对应 Embedding 模型的语义编码
- `generate` 而不是 `chat`：对应 LLM 的自回归生成
- `logits`：单独拆出来，对应实习中的 `hbm_runtime.run()` Prefill 阶段，用于 OpenCompass 评测

**为什么返回值都是 numpy array？** 这是刻意向 `hbm_runtime` 的接口风格靠拢。面试时你可以说："我在实习中发现，端侧推理框架（hbm_runtime、ONNX Runtime）都以 numpy array 作为中间表示。所以这个抽象层也沿用了同样的设计，这样切换到手机端 ONNX 推理时，上层代码完全不用动。"

**为什么 `generate` 的 `max_tokens` 默认 512？** 对于个人记忆助手这个场景，答案通常比较简短（几十到几百 token），不需要长文生成。512 是一个安全的默认值，既不浪费推理时间，也覆盖了大多数问答需求。

### 4.3 LocalSimulateBackend 的实现要点

```python
class LocalSimulateBackend(InferenceBackend):
    def __init__(self, embedding_model_name, llm_model_name, chunk_size, device):
        self._embedding_model = SentenceTransformer(embedding_model_name, device=device)
        self._llm_model = AutoModelForCausalLM.from_pretrained(
            llm_model_name, torch_dtype=torch.float16, trust_remote_code=True
        ).to(device)
```

**关键设计点：**

1. **`torch.float16` 只在 CUDA 设备上使用。** RTX 4060 支持 FP16 推理，可以节省一半显存。但 CPU 上必须用 FP32，因为没有硬件加速。

2. **`trust_remote_code=True`**：Qwen 系列的 tokenizer 需要这个参数，因为它有自定义的 tokenization 逻辑。

3. **延迟记录：** 每次推理都记录 `_last_embed_latency_ms` 和 `_last_generate_latency_ms`。这些数据是 Benchmark 的原材料，也是面试时可以展示的定量数据。

### 4.4 MobileBackend 骨架的设计意图

```python
class MobileBackend(InferenceBackend):
    def __init__(self, ...):
        raise NotImplementedError(
            "MobileBackend 需要在 android 设备上运行。"
            "PC 开发阶段请使用 LocalSimulateBackend。"
        )
```

`MobileBackend` 现在只是一个骨架（抛出 NotImplementedError），但它在架构中的存在很重要：

- **告诉面试官你有端侧部署的意识。** 注释中详细写了 Android ONNX Runtime 和 iOS CoreML 的部署流程。
- **抽象层已经预留了切换点。** 将来真正做手机端部署时，只需要实现 MobileBackend 的三个抽象方法，上层所有 Agent 代码不用改一行。
- **和 hbm_runtime 形成类比。** .hbm 编译 ⇄ .onnx 导出（离线步骤），hbm_runtime.run() ⇄ ORT.Run()（推理调用），Chunk=512 约束在手机端同样适用。

**面试引导话术：** "这个 MobileBackend 骨架虽然现在不能用，但它体现了我对端侧推理的理解——不同的推理框架有不同的部署流程，但抽象出统一的推理接口后，上层业务逻辑就与具体的推理框架解耦了。这和我在实习中做 S600 芯片推理评测时，hbm_runtime 的设计理念是一致的。"

- [ ] 能说出 InferenceBackend 三个抽象方法各自对应 hbm_runtime 的哪个操作
- [ ] 能解释 `chunk_size=512` 为什么在端侧场景下是一个关键约束参数
- [ ] 能解释 `MobileBackend` 骨架在架构中的价值（即使暂时不能运行）
- [ ] `pytest tests/test_backend.py -v` 全部通过

⚠️ **常见误区：**
- **"抽象层就是多写了一个 interface"** → 不是。抽象层的价值在于切换后端时业务代码零改动。如果你把 `backend.generate()` 直接写在 Agent 代码里，换到 ONNX Runtime 时要改 10+ 个文件。
- **"MobileBackend 不能用，没价值"** → MobileBackend 的价值是告诉面试官你**考虑了**端侧部署。代码里的注释详细写了 ONNX/CoreML 的部署流程——这比"我能跑"更有说服力，因为体现了架构前瞻性。
- **"三个抽象方法是不是随便定义的"** → 不是。embed/generate/logits 分别对应 Embedding 推理、自回归生成、评测 Logits——这是从 hbm_runtime 的实际接口中抽象出来的。面试时主动引出实习经验的关键入口。

## 第 5 节：模块二——数据摄取管线

> **学习目标：** 理解真实世界数据的 messy 本质，学会设计鲁棒的数据解析器。

### 5.1 为什么多数 AI 项目没有数据管线？

因为多数项目用现成的数据集，CSV 一读就开始跑模型。但实际上，**真实 AI 应用开发中，60% 的工作量在数据处理上。**

PocketMemory 的场景决定了我们必须自己写数据摄取：

1. **微信聊天记录**没有标准格式。桌面版导出的 TXT 文件格式多变。
2. **个人笔记**是 Markdown 和 PDF 混用的。
3. **时间戳是关键元数据**，决定了检索的时效性。

### 5.2 WeChat 解析器的设计

微信桌面版导出的格式大致是：
```
2026-03-15 14:30:22 张三
周末有空吗？一起吃个饭？

2026-03-15 14:31:05 我
好的，周六中午怎么样？
```

看起来简单，实际有很多坑：

**坑 1：多行消息。** 用户可能一次发送多行文本（比如发了一段长文字、或者复制粘贴了文章），这些行的开头不会匹配到"日期 时间 发送者"的格式。解析器需要把这些续行归到前一条消息。

**坑 2：系统消息。** "张三邀请李四加入了群聊"、"你撤回了一条消息"——这些消息没有对话价值，需要识别并过滤。

**坑 3：媒体标记。** "[图片]"、"[文件]"、"[视频]"——这些消息有元数据但没有文本内容。对纯文本检索来说没有用，但可以作为消息类型的标记。

**坑 4：日期格式变体。** 不同版本的微信导出可能用 `YYYY-MM-DD`、`YYYY/MM/DD`、或 `MM-DD`。解析器要做归一化。

**设计原则：松进严出。** 解析时尽可能宽松（不因一个格式变异就崩溃），但输出时严格分类（ChatMessage 有明确的 msg_type）。

```python
@dataclass
class ChatMessage:
    timestamp: datetime
    sender: str
    content: str
    msg_type: str  # text | system | image | file | voice
    chat_type: str  # private | group
    chat_name: str
```

**为什么用 dataclass 而不是 dict？** 类型安全。16 个字段的 dict 在传递过程中容易拼错 key，dataclass 有 IDE 自动补全和类型检查。

### 5.3 对话感知分块器

普通的 RAG 系统按固定 token 数切块。但对聊天记录来说，这样做会破坏对话的连贯性：

```
不好的分块：                         好的分块（对话感知）：
[Block 1]                            [Block 1]
张三: 周末有空吗？                    张三: 周末有空吗？一起吃饭？
我: 好的                           我: 好的，周六中午怎么样？
张三: 去上次那家火锅店吧               张三: 去上次那家火锅店吧，渝味火锅

[Block 2]                            [Block 2]
一起吃饭？                           李四: 项目方案发你了
张三: 周六中午怎么样？
```

**分块策略：**
1. **先按时间分组（30 分钟间隔 = 新对话）**——因为现实中一次微信对话通常不会中断超过 30 分钟
2. **每个对话内部再按 token 限制切块**
3. **切块时保留最后一条消息作为重叠**——防止关键信息被切断在两块的边界

**为什么是 30 分钟？** 这是一个工程近似值。微信对话的典型节奏是"几秒到几分钟一条消息"，30 分钟足够覆盖一次完整对话。如果你发现自己的聊天记录有不同的节奏，可以调整 `gap_minutes` 参数。

### 5.4 索引器设计

```python
class Indexer:
    def index(self, chunks: List[DocumentChunk]) -> int:
        # 去重（跳过已索引的 chunk_id）
        new_chunks = [c for c in chunks if c.chunk_id not in self._indexed_hashes]
        # 写入向量库 + BM25
        self._vector_store.add_chunks(new_chunks)
        self._bm25.index([c.content for c in new_chunks])
```

**为什么同时写两个索引？** 向量检索擅长语义匹配（"好吃的火锅"和"川菜推荐"），BM25 擅长关键词匹配（"渝味火锅"精确查找）。两者互补，通过 RRF 融合后效果最好。

## 第 6 节：模块三——RAG 检索引擎

> **学习目标：** 理解混合检索的原理，掌握向量检索和关键词检索各自的适用场景。

### 6.1 向量检索 vs 关键词检索：互补而非替代

| 维度 | 向量检索 (ChromaDB) | 关键词检索 (BM25) |
|------|---------------------|-------------------|
| 原理 | 语义相似度（余弦距离） | 词频-逆文档频率 |
| 擅长 | "好吃的火锅"找"川菜推荐" | "渝味火锅"精确匹配 |
| 弱项 | 专有名词、缩写、数字 | 同义词、改写表达 |
| 中文支持 | 依赖 Embedding 模型 | 依赖分词器 |

**为什么两者都有用？** 考虑这个查询："张三推荐的那家火锅店叫什么？"

- 向量检索：理解"火锅店"和"餐厅"的语义关系，找到相关对话
- BM25：精确匹配"张三"、"火锅"这些关键词

两者结合（RRF 融合），召回率比单用任一种高 10-15%。

### 6.2 OnDeviceEmbeddings 的设计

```python
class OnDeviceEmbeddings:

        # LangChain 兼容接口

    def embed_query(self, text: str) -> List[float]:

    def embed_raw(self, texts: List[str]) -> np.ndarray:

```

**为什么提供三种接口？**
- `embed_documents` / `embed_query`：LangChain 兼容格式，方便和生态集成
- `embed_raw`：绕过 LangChain 的开销，直接拿到 numpy array 做相似度计算（缓存层需要）

**`batch_size=32` 是怎么定的？** BGE-small 的 Embedding 推理很轻量，32 条文本一批不会 OOM（8GB 显存绰绰有余），同时保持了较高的 GPU 利用率。

### 6.3 RRF 融合——数学原理

RRF（Reciprocal Rank Fusion）公式：

```
RRF_score(d) = Σ 1/(k + rank_i(d))
```

其中：
- `k` 是平滑常数（我们取 60）
- `rank_i(d)` 是文档 d 在第 i 路检索结果中的排名（从 1 开始）

**为什么用 RRF 而不是简单的分数加权？** 因为向量检索的余弦距离和 BM25 的 TF-IDF 分数不在同一个量级上，不能直接比较或加权。RRF 只关心排名，天然实现了分数的归一化。

**k=60 是怎么来的？** 来自 TREC（国际文本检索会议）的多轮实验验证。k 越小，高排名文档权重越大；k 越大，越趋近于简单平均。60 在大多数数据集上表现最好。

## 第 7 节：模块四——时间感知 + 实体提取

> **学习目标：** 理解为什么个人记忆场景需要时间感知，以及如何用轻量级方法做实体识别。

### 7.1 时间的本质是信息衰减

在个人记忆场景中，时间有三个作用：

1. **衰减权重。** 昨天的对话比去年的对话更可能被用户查询。
2. **时间约束过滤。** 用户说"上个月"，意味着只需要检索最近 30 天的数据。
3. **答案的时间标注。** "根据你们 3 月 15 日的聊天记录"——这个时间标注让答案可信度大增。

**时间衰减公式：**

```
final_score = rrf_score × exp(-λ × days_since)
```

其中 `λ = ln(2) / half_life_days`

**为什么是指数衰减而不是线性？** 因为人类对记忆的遗忘是指数级的（艾宾浩斯遗忘曲线）。30 天的半衰期意味着：一个月前的对话权重减半，两个月前的减为 1/4。

**半衰期 30 天是怎么定的？**

说实话：这是一个可以调的参数，30 天是一个合理的初始值。你可以在 Benchmark 上用不同半衰期（15 天、30 天、60 天）跑一遍，对比 Recall@5——面试时这就是你的"参数调优实验"。

### 7.2 实体提取：为什么不用 NER 模型？

加一个 NER 模型（如 bert-base-chinese-ner）会额外占用 ~400MB 显存。在 8GB 显存下，已经装了 LLM（~3GB）+ Embedding（~100MB），再加 NER 会挤占 KV Cache 空间。

**我们选择了一个轻量级方案：**

1. **人名识别：** 百家姓字典 + jieba 词性标注（nr 标签）+ 正则
2. **日期识别：** 正则匹配 YYYY-MM-DD、MM月DD日、相对时间（昨天/上周）
3. **主题关键词：** TF-IDF，取 Top 5 per chunk

**这个方案的局限和优势：**
- 局限：专有名词（公司名、产品名）识别率低，"今天"可能被误识别为日期实体
- 优势：零显存开销、毫秒级速度、对于个人聊天场景（人名+日期为主）够用

**面试时怎么说：** "我选择了一个轻量级的实体提取方案而不是加载额外的 NER 模型，因为 8GB 显存约束下，每一 MB 都要规划用途。对于个人记忆助手这个场景，需要识别的实体主要是人名和日期，规则+词典的方案能达到 80% 以上的识别率，但节省了 400MB 显存。当然如果用更强的硬件，换上 NER 模型可以进一步提升实体识别的精确度。"

### 7.3 实体感知检索

```
查询 → 提取实体 → 混合检索（向量+BM25）→ RRF融合 → 实体加权 → 排序返回
```

**实体加权怎么算？**

```python
boost = 1.0
# 每个人名命中：×1.2
boost *= 1.2 ** people_match_count
# 每个主题命中：×1.1（权重较低）
boost *= 1.1 ** topic_match_count
```

**为什么人名加权 1.2 而主题只 1.1？** 因为查询中的"张三"意味着用户明确想找和这个人相关的对话。人名是强信号。而主题关键词（如"火锅"）可能出现在不直接相关的对话中，是弱信号。

## 第 8 节：模块五——Agent 编排：MemoryJudge

> **学习目标：** 理解 Agent 图编排的核心概念，掌握从 Evidence Judge 到 MemoryJudge 的简化逻辑。

### 8.1 先理解原来 Evidence Judge 为什么"重"

原来的教学文档中，Evidence Judge 有四步：

1. **相关性打分**（LLM 调用）：对每个检索结果用 LLM 打分（0-100）
2. **内容去重**（MD5 哈希）
3. **信源标注**（trusted/untrusted/uncertain）
4. **充足性判断**（≥1 个 trusted 或 ≥3 个任意）

问题在于**第 1 步和第 3 步**：

- 假设检索到 10 个文档块，每个都要用 LLM 单独打分 → 10 次 LLM 调用
- 信源标注也需要 LLM 判断来源可信度 → 额外调用

对于一个 4B 参数的量化模型，每次 LLM 调用需要 200-500ms。10 次调用就是 2-5 秒。再加上生成答案的时间，总延迟可能超过 8 秒——用户早就不耐烦了。

**但更根本的问题是：这些步骤设计的初衷是什么？** Evidence Judge 是为**不可信数据源**设计的——网页搜索可能返回虚假信息、PDF 文档可能来自不可靠的作者。所以需要逐一验证和标源。

个人记忆场景中，所有数据都是用户自己的聊天记录——天然可信。

### 8.2 MemoryJudge 的设计思路

```python
# Step 1: 组合评分 = 0.6×语义 + 0.3×时间 + 0.1×实体
composite = 0.6 * semantic + 0.3 * temporal + 0.1 * entity_bonus

# Step 2: 对话去重（前100字符 MD5）
# Step 3: 充足性（≥2个唯一会话 或 ≥3个高置信度chunk）
# Step 4: Reflect（不足时生成1个补充查询，最多1次）
```

**为什么语义权重 0.6 最高？** 语义相关性是检索质量的根本。如果检索到的内容和查询毫无关系，时间再新也没用。

**为什么时间占 0.3？** 对于记忆助手来说，"昨天发生的"比"一年前的"更有价值。但时间不是决定性因素——有时候用户就是想找很久以前的某条重要信息。

**实体只占 0.1？** 实体匹配是锦上添花，不是决定因素。查询中提到"张三"不一定意味着只想要张三的消息（可能是问"张三和李四上次讨论了什么"）。

### 8.3 Reflect 为什么最多 1 次？

Reflect 循环是有成本的：每次补充搜索意味着重新跑一轮检索 + 判断。如果无限循环，延迟会爆炸。

1 次是一个工程权衡：
- 第一次检索可能遗漏（因为查询表达不精确）
- 补充搜索可以弥补（用补充查询找回遗漏的结果）
- 如果补搜后还不够 → 直接生成答案并告知信息不足

**面试时怎么回答"为什么不多搜几次"：** "Reflect 循环是典型的搜索效率和信息增益的平衡问题。第一次补充搜索通常能带来最大的信息增量（找回遗漏的关键信息），但第三次、第四次搜索的边际收益递减——补搜的结果往往和前两轮高度重叠。同时，每次补搜增加 500-800ms 延迟。所以 1 次补搜是延迟和召回率的较优平衡点。"

### 8.4 LangGraph 图编排

```python

    "retrieve": "retrieve",    # memory_lookup / knowledge_lookup

    "end": END,                # refuse
})

# 判断后条件边：充足→生成，不足→补搜→回到检索
workflow.add_conditional_edges("judge", _route_after_judge, {
    "generate": "generate",
    "reflect": "retrieve",     # ← 补搜循环
    "end": END,
})

workflow.add_edge("generate", END)
```

**图的执行流程：**

```
START
  │
  ▼
Router ──(general)──→ Generate ──→ END
  │
  ├──(refuse)──→ END
  │
  └──(memory_lookup/knowledge_lookup)
       │
       ▼
    Retrieve
       │
       ▼
    Judge ──(sufficient)──→ Generate ──→ END
       │
       └──(insufficient, <2次)
            │
            ▼
         Reflect → (补充查询) → Retrieve (回到检索)
```

**条件边 vs 普通边：**
- 普通边（`add_edge`）：固定跳转，A 之后总是到 B
- 条件边（`add_conditional_edges`）：根据 state 内容动态决定跳转目标

**`add_messages` reducer 是什么？** 在 `AgentState` 中：

```python
messages: Annotated[list, add_messages]
```

`add_messages` 是 LangGraph 内置的 reducer，当多个节点都往 `messages` 追加内容时，不会互相覆盖，而是合并成一个列表。这样每一步的日志（router 的输出、retriever 的统计、judge 的判断）都能完整保留在 state 里。

- [ ] 能解释组合评分公式中各权重的设计理由

⚠️ **常见误区：**
- **"Agent 图就是画几个框和箭头"** → 关键在条件边设计。Router→Retrieve 是条件边（根据意图决定路径），Retrieve→Judge 是普通边（固定跳转）。把条件边画成普通边，Reflect 循环就失效了。
- **"组合评分公式是拍脑袋定的权重"** → 语义 0.5 最高因为相关性是根基；时间 0.2 是记忆场景的差异化维度；实体 0.1 是锦上添花。面试官追问时能说出每个权重的理由。
- **"Reflect 为什么不多循环几次"** → 边际收益递减——第一次补搜的信息增量最大，第二次开始结果高度重叠。加上每次补搜 +500ms 延迟，1 次是最优平衡点。

## 第 9 节：模块六——API 服务 + 缓存

> **学习目标：** 理解 SSE 流式传输的适用场景，掌握双层缓存的设计思路。

### 9.1 为什么用 SSE 而不是 WebSocket？

这两种技术都能做服务端推送，但适用场景不同：

| 维度 | SSE | WebSocket |
|------|-----|-----------|
| 通信方向 | 服务端 → 客户端（单向） | 双向 |
| 协议 | HTTP（复用现有基础设施） | 独立协议（需要升级握手） |
| 浏览器兼容 | 原生 EventSource API | 需要额外库 |
| 自动重连 | 内置 | 需要手动实现 |

对于 PocketMemory 的场景：客户端发送一个查询请求，服务端逐步返回"路由→检索→判断→生成"的中间状态。这是典型的单向流，SSE 完全够用，而且比 WebSocket 简单。

### 9.2 双层缓存的设计思想

```
用户查询 → L1 精确匹配（MD5 哈希）→ 命中 → 直接返回
              ↓ 未命中
         L2 语义匹配（余弦相似度 > 0.95）→ 命中 → 写入 L1 → 返回
              ↓ 未命中
         Agent 全链路推理 → 写入 L1 + L2 → 返回
```

**为什么要两层？**

- **L1（精确缓存）**：相同查询直接命中。TTL 设 5 分钟——因为用户短时间内可能重复问同一个问题（"再看一下那家火锅店叫什么"）。
- **L2（语义缓存）**：相似查询命中。"张三推荐了哪家火锅店"和"张三说的那个火锅店叫什么"本质上是一个问题。余弦相似度阈值 0.95 确保语义足够接近。

**TTL 为什么不同？**
- L1 TTL=5 分钟：精确匹配窗口短，避免用户改了数据后缓存还返回旧的
- L2 TTL=30 分钟：语义匹配窗口长，因为个人聊天数据变化不频繁

**缓存键的设计：** L1 用 `MD5(query)` 做 key，O(1) 查找。L2 用 `cosine_similarity(query_embedding, stored_embedding)`，O(n) 但缓存的条目通常不超过 128 条，可以接受。

## 第 12 节：面试深挖准备

### 必问 13 题

**Q1：为什么设计 InferenceBackend 抽象层？**

不是为了抽象而抽象。我在实习中发现，S600 芯片的 hbm_runtime 和手机端的 ONNX Runtime 虽然推理框架不同，但有一个共同点：都以 numpy array 作为中间表示。所以这个抽象层的三个方法（embed/generate/logits）都是 numpy array 输入输出。上层 Agent 代码不关心底层是 hbm_runtime 还是 ONNX Runtime。换芯片时，业务代码零改动。

**Q2：MemoryJudge 和 Evidence Judge 的区别？**

Evidence Judge 是为不可信数据源设计的（网页搜索），需要四步验证（打分+去重+信源+充足性）。个人记忆场景中，数据都是用户自己的聊天记录——先天可信。所以 MemoryJudge 简化为组合评分公式（0.6×语义+0.3×时间+0.1×实体），LLM 调用从 ~12 降到 ~2 per query，延迟从约 5s 降到约 1.5s。

**Q3：RRF 融合的 k 为什么是 60？**

来自 TREC 实验结论，k=60 在大多数数据集上最稳定。k 越小，排名靠前的文档权重越大；k 越大越趋于平均。60 是经过验证的平衡点。

**Q4：W4A8 量化对 Agent 推理精度的影响？**

我在实习中做过 Qwen3-4B 的 W4A8 量化，PPL 损失控制在 2-5%。对于记忆助手这个场景，检索增强了上下文，降低了模型本身生成的压力——模型只需要"整理和复述"检索到的信息，而不是凭空生成。所以量化损失对最终答案质量影响很小。

**Q5：ChromaDB 能支撑多大知识库？**

ChromaDB 用 HNSW 索引，10 万条文档内查询延迟 <50ms。个人记忆场景（聊天记录+笔记），一个人几年累积的数据大概几千到几万条，绰绰有余。如果要做企业级多用户版本，需要迁移到 Milvus。

**Q6：Reflect 补搜会不会无限循环？**

不会。最多 1 次。原因有二：一是边际收益递减（第二次补搜的信息增量和第一次高度重叠），二是延迟控制（每次补搜 +500~800ms）。1 次是召回率和体验的平衡点。

**Q7：时间衰减为什么是指数而不是线性？**

因为符合艾宾浩斯遗忘曲线——人类对记忆的遗忘规律是指数级的。线性衰减意味着"一年前的对话权重为零"，但实际上用户可能确实需要找一年前的重要信息。指数衰减更柔和：一年前的对话权重约为 0.0004，仍然有机会被检索到。

**Q8：与 Dify/RAGFlow 框架的区别？**

Dify 和 RAGFlow 是通用 RAG 平台，面向企业场景，调云端 API。PocketMemory 做的是端侧个人场景，**本地推理 + 离线运行**。最大区别在于我对硬件约束的考量——8GB 显存决定了模型大小、是否加 NER 模型、参数量的选择——这些都是从端侧经验出发做的决策，而不是在无约束的云环境下做开发。

**Q9：怎么把项目产品化？**

三步走：第一，完善数据源接入（微信/钉钉/飞书/iMessage），覆盖更多用户场景。第二，做 IM 自动化（自动标签、自动摘要、定期回顾），从"被动搜索"到"主动服务"。第三，真机端侧推理（模型 ONNX 导出 + 移动端 ORT Runtime），做到完全离线。

**Q10：这个项目最大的技术挑战是什么？**

不是某个具体算法，而是**8GB 显存约束下的系统工程**——如何在一个小显存上同时跑 Embedding + LLM + 向量索引，同时保证延迟在 3 秒以内。每个技术决策都是在这个约束下做的：选 BGE-small 而不是更大的 Embedding、不加 NER 模型、MemoryJudge 用公式而不是多轮 LLM 调用。这种"约束驱动设计"是端侧开发的核心能力。

**Q11：你的模型选择策略？**

开发阶段用小模型（Qwen2.5-1.5B）快速迭代，验证功能和流程。生产环境用量化后的 4B 模型（Qwen3-4B W4A8）提升效果——这正好用上了我实习中的 LLMC 量化经验。选型策略是"快速验证 + 逐步升级"，而不是一开始就上大模型。

**Q12：为什么选 BGE-small-zh-v1.5 做 Embedding？**

三个理由：尺寸小（~100MB，不挤占 LLM 的显存空间）、中文效果好（在 C-MTEB 中文榜单上排名靠前）、接口兼容（sentence-transformers 库原生支持）。如果要升级，可以用 Qwen3-Embedding-0.6B 量化版，正好是实习中做过的优化。

**Q13：手机端跑 4B 模型能行吗？**

量化后（W4A8）约 2.3GB，骁龙 8 Gen 3 上首 token 约 300ms。加上我们的检索增强，大部分查询不需要模型自己"想"，只需要"整理"，所以 token 数少、延迟可控。

### 5 个场景追问

**追问 1：如果用户换了手机，索引怎么迁移？**

ChromaDB 的数据存在 `./data/chroma_db/` 目录下，是一个文件夹。迁移 = 复制这个文件夹到新手机 + 重新加载。后续可以做云备份（加密后的索引文件同步到 iCloud/Google Drive），但保持客户端解密。

**追问 2：微信聊天记录里有图片和语音怎么办？**

当前版本只处理文本。扩展方向：图片用端侧 OCR/图像描述模型（如 BLIP）转文字后索引；语音用 Whisper 转文字。这些模型都有量化版本，可以在端侧运行。

**追问 3：数据隐私具体怎么保障？**

Three-layer: 传输层（所有数据通过本地 HTTP，不出设备）、存储层（ChromaDB 本地持久化，不上传）、模型层（推理在本地 GPU/NPU 完成，不调云端 API）。即使手机丢失，数据也是加密存储的。

**追问 4：系统延迟能进一步优化吗？**

有几个方向：提示词缓存（相同 system prompt 的 KV Cache 复用）、投机解码（用更小的 draft model 加速生成）、模型量化到更低精度（INT8）。Target: end-to-end < 2s on RTX 4060.

**追问 5：如果让你重做这个项目，你会改什么？**

第一，Embedding 模型从 BGE-small 换成 Qwen3-Embedding-0.6B 量化版（做得更"端侧"）。第二，加一个轻量级的 Cross-Encoder 做精排（如果能解决显存问题）。第三，Agent 图加一个 Fallback 节点——当检索不到任何相关信息时，不是直接说"不知道"，而是主动引导用户提供更多线索。

### 新功能面试深挖（Hook / 用户画像 / 数据源 / Prompt 管理器）

**Q14：你的 Hook 系统和 MewCode 的生命周期钩子有什么区别？**

MewCode 的 Hook 是为 Coding Agent 设计的——围绕"执行工具"这个动作（工具调用前后、会话开始结束）。我的 Hook 是为知识型 Agent 设计的——4 个挂载点围绕"检索信息"这个动作（路由前、检索前、检索后、生成后）。

设计理念一样（对扩展开放，对修改关闭），但挂载点的选择必须匹配 Agent 的执行流程。关键是：**Hook 失败不阻塞主流程**——用户画像更新失败不应该导致查询失败。

**Q15：用户画像为什么不存 ChromaDB 而是存 JSON？**

ChromaDB 存的是"事实"（张三 3 月 15 日说了什么），用户画像存的是"元认知"（用户更关心张三而不是李四）。前者是向量检索的数据，后者是单条记录——一个用户一个画像，不需要语义搜索。

工程判断：为正确的数据选正确的存储。JSON 文件比 ChromaDB client+collection 轻量得多，`json.dump()` 一行搞定。

**Q16：微信导出 TXT 和真正数据库（EnMicroMsg.db）有什么区别？为什么不直接读数据库？**

微信桌面版导出 TXT 是纯文本格式，解析简单（按行读+时间戳正则）。真正的微信数据库是 SQLCipher 加密的 SQLite 文件，需要获取解密密钥（MD5(IMEI+UIN) 前 7 位），需要 root 权限。

当前支持 TXT 导出作为 MVP，体现"先跑通再优化"的工程思维。后续接入加密数据库是 Roadmap——这和实习中做模型量化的思路一样：先用简单方案验证流程，再用复杂方案提升效果。

**Q17：你的 Prompt 管理和直接把 Prompt 写死在代码里有什么区别？**

写在代码里：改了要重启，没有版本记录，面试无法展示迭代。
我的方案：YAML 集中管理，`prompts.reload()` 热加载，每个 Prompt 带 `version` 和 `changelog`。面试时直接打开展示 Router prompt 从 v1（JSON 输出，70% 成功）到 v2（单字+兜底，100% 成功）的演进过程。**这证明 Prompt 是你迭代出来的，不是 AI 一次性生成的。**

**Q18：数据源管理器的增量索引怎么实现？怎么避免重复索引？**

`SourceManager._indexed_files` 字典：key 是文件路径，value 是索引时间+片段数。`ingest_new()` 扫描到文件后先检查是否在字典中——是则跳过，否则索引并记录。对 200 条消息无所谓，但对 10 万条消息就是 3 分钟 vs 3 秒的差异。

### 新功能面试深挖（Hook / 用户画像 / 数据源 / Prompt 管理）

**Q14：你的 Hook 系统和 MewCode 的生命周期钩子有什么区别？**

MewCode 的 Hook 是为 Coding Agent 设计的——工具调用前后、会话开始结束等。我的 Hook 是为知识型 Agent 设计的——4 个挂载点分别是路由前、检索前、检索后、生成后。区别在于：MewCode 的 Hook 围绕"执行工具"这个动作，我的 Hook 围绕"检索信息"这个动作。

设计理念是一样的——对扩展开放，对修改关闭。但挂载点的选择必须匹配 Agent 的执行流程。比如我的 `post_retrieve` hook 天然适合做时间衰减，`post_generate` hook 天然适合更新用户画像。

**Q15：用户画像和 ChromaDB 里的聊天记录有什么区别？为什么不直接存 ChromaDB？**

ChromaDB 存的是"事实"——张三在 3 月 15 日说了什么。用户画像存的是"元认知"——用户更关心张三（而不是李四）、更关注火锅店（而不是天气）。前者是数据，后者是洞察。

用 JSON 而不是 ChromaDB 存储用户画像的理由：用户画像是单条记录（一个用户一个画像），不需要向量检索。JSON 可以直接 `json.dump()` 保存，比 ChromaDB 的 client + collection 轻量得多。这也是工程判断——为正确的数据选正确的存储。

**Q16：微信导出 TXT 和真正的微信数据库（EnMicroMsg.db）有什么区别？为什么不直接读数据库？**

微信桌面版导出的 TXT 是纯文本格式，解析简单（按行读 + 正则匹配时间戳）。真正的微信数据库是 SQLCipher 加密的 SQLite 文件（EnMicroMsg.db），需要获取解密密钥（MD5(IMEI + UIN) 的前 7 位）。

不直接读数据库的原因：
1. 需要手机 root 权限才能拿到数据库文件
2. 需要 IMEI + UIN 推导解密密钥，涉及隐私
3. 不同微信版本的数据库 schema 可能不同

当前设计支持 TXT 导出格式作为 MVP，后续接入数据库是 Roadmap 的一步——这正好体现"先跑通再优化"的工程思维。

**Q17：你的 Prompt 管理和直接把 Prompt 写在代码里有什么区别？**

把 Prompt 写在代码里：改了要重启服务，没有版本记录，面试时无法展示迭代过程。

我的 Prompt 管理器：YAML 文件集中管理，`prompts.reload()` 热加载（改 YAML 不需要重启），每个 Prompt 带 `version` 和 `changelog` 字段。面试时可以直接打开展示 Router prompt 的演进——v1 用 JSON 输出但小模型不稳定（70% 成功），v2 改单字输出加关键词兜底（100% 成功）。

**Q18：数据源管理器的增量索引是怎么实现的？**

通过文件路径去重。`SourceManager._indexed_files` 是一个字典，key 是文件路径，value 是索引时间+片段数。`ingest_new()` 方法会检查每个扫描到的文件是否已经在 `_indexed_files` 中——如果是就跳过，不是就索引并记录。

这个设计避免了每次启动都重新索引所有文件。对于 200 条消息的聊天记录来说差异不大（几秒 vs 几秒），但对于 10 万条消息的场景来说就是 3 分钟 vs 3 秒的差异——增量索引是关键优化。

## 第 13 节：新增功能面试追问 🆕

> 以下问题针对本轮迭代新增的 Hook 系统、用户画像、数据源管理器和 Prompt 管理器。

**Q14：Hook 系统是做什么的？为什么需要它？**

"我在 Agent 执行的 4 个关键节点预留了挂载点——pre_route、pre_retrieve、post_retrieve、post_generate。新功能通过注册 hook 函数接入，不需要改核心代码。比如用户画像功能就是通过 pre_retrieve 和 post_generate 两个 hook 接入的。这体现了'对扩展开放，对修改关闭'的设计原则。Hook 失败也不会阻塞主流程——用户画像更新失败不应该导致查询失败。"

**Q15：用户画像怎么工作的？和 MewCode 的跨会话记忆有什么不同？**

"用户画像是通过 post_generate hook 自动记录的——每次查询后，系统用 jieba 分词提取查询中的关键词，更新高频词统计和联系人关注度。10 次查询后，用户问'上次那个火锅店'时，系统已经从画像知道'火锅店'是高频话题。和 MewCode 的跨会话记忆是同一类功能，但我把它适配到了记忆检索场景——MewCode 记的是'用户喜欢怎么写代码'，我记的是'用户经常搜什么话题、关注哪些人'。"

**Q16：数据源管理器支持哪些格式？怎么检测微信导出格式？**

"支持微信桌面版导出 TXT、Markdown 笔记、PDF 文档。微信 TXT 的格式检测用启发式规则——读取文件前 10 行，检查有多少行匹配 'YYYY-MM-DD HH:MM:SS 发送者' 模式。≥2 行匹配就判定为微信格式。增量索引通过文件路径去重，不会重复处理。前端采用两栏布局，左侧 260px 面板常驻显示数据源，分为「本地文件」和「从微信导入」两个区域，每个区域显示文件数和片段数。"

**Q17：Prompt 管理器怎么做版本管理的？**

"所有 Prompt 从代码中抽离到 YAML 配置文件，每个文件带版本号和 changelog。比如 Router prompt 的 changelog 里记录了 v1 用 JSON 输出但 1.5B 模型不稳定（70% 成功率），v2 改成单字输出加关键词兜底（100% 成功率）。面试时可以直接打开展示 Prompt 的演进过程。另外支持热加载——改 YAML 不需要重启服务。"

**Q18：你的项目和 MewCode Agent 有什么区别？**

"MewCode 是工具型 Agent——它的核心循环是'思考→调工具→看结果→再思考'，工具是文件操作和命令执行。PocketMemory 是知识型 Agent——核心循环是'理解意图→检索记忆→判断充分→生成回答'，工具是检索算子和时间衰减。这是场景驱动的架构差异化——不是所有 Agent 都需要 Tool Calling 和权限系统，我的场景是'找信息'而不是'写代码'。"

## 第 14 节：学完之后的扩展路线 🟢

> 项目跑通了，面试题背熟了。下一步可以做什么？以下 5 条路线按复杂度递增，选一条深入可以让你在面试中多聊 10 分钟。

### 路线 1：接入真实微信数据（2 小时，最简单）

当前用 demo 数据。替换成你自己的：

```bash
# 1. 微信桌面版 → 右键聊天 → 导出聊天记录 → 得到 张三.txt
# 2. 放到 data/raw/ 目录
cp ~/Desktop/张三.txt data/raw/

# 3. 启动服务
uv run python scripts/run_demo.py --serve

# 4. 浏览器打开 → 数据源管理 → 扫描新文件
# 5. 问你自己的问题："上周我和张三聊了什么？"
```

**面试附加值：** "我用自己真实的微信聊天记录测试过，195 条模拟数据变成了 2000+ 条真实对话。"

### 路线 2：量化模型替换（1 天，最高面试价值）

当前跑的是 Qwen2.5-1.5B（开发验证用）。替换成你实习中学到的 LLMC W4A8 量化模型：

1. 下载 Qwen3-4B-Instruct
2. 用 LLMC 做 W4A8 量化（QuRot + LWC + GPTQ）
3. 替换 `settings.py` 中的 LLM 模型路径
4. 重新跑 Benchmark，对比 1.5B vs 4B-W4A8 的延迟和答案质量

**面试附加值：** "我把实习中的 W4A8 量化技术应用到了这个项目，4B 量化模型在 8GB 显存下跑通，PPL 损失 <3%。"

### 路线 3：三层记忆架构（3 天，架构深度）

目前的记忆系统是"一层"——所有聊天记录平等对待。加上之前在 Claude Code 对话中讨论的三层架构：

- **Layer 1 感官记忆：** 24 小时内的消息权重最高，自动衰减
- **Layer 2 短期记忆：** 被搜索过/被回复过的消息自动提升重要性
- **Layer 3 长期记忆：** 多段相关对话自动摘要合并成"知识点"

**面试附加值：** "当前版本是记忆系统的最小可行版本。如果要做得更像人类记忆，我会加入三层架构和基于访问频率的强化机制。但 V1 的定位是先验证端侧全链路可行性。"

### 路线 4：直接读微信加密数据库（1 周，硬核挑战）

跳过"手动导出 TXT"，直接读微信的 SQLCipher 加密数据库（EnMicroMsg.db）：

1. 获取数据库文件（需要手机 root 或 PC 微信目录）
2. 推导解密密钥（MD5(IMEI + UIN) 的前 7 位）
3. 用 sqlcipher 库解密并读取消息表
4. 适配不同微信版本的 schema 差异

**面试附加值：** "当前版本支持 TXT 导出作为 MVP。后续计划直接接入加密数据库——这需要解决 SQLCipher 解密、schema 适配、权限获取三个问题，和实习中做的底层适配工作是同一类技术挑战。"

### 路线 5：真机端侧部署（2 周，终极目标）

把整个系统从 PC 搬到手机上：

1. Embedding 模型：ONNX 导出 → ONNX Runtime Mobile 加载
2. LLM：W4A8 量化 → ONNX 导出 → 手机上自回归推理
3. ChromaDB：替换为 SQLite + 暴力搜索（数据量小时够用）
4. API 服务：PC 端 FastAPI → 手机端本地 HTTP Server

**面试附加值：** "InferenceBackend 抽象层的设计就是为了这一步——上层 Agent 代码不用改，只需要实现 MobileBackend 的三个抽象方法。"
